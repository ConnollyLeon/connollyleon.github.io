# 科技新闻：AWS发布Context知识图谱服务，GitHub AI代理危机致Microsoft借助AWS基础设施

企业AI基础设施领域迎来多项重大发展。AWS发布Context知识图谱服务，为AI代理提供自动学习的上下文层；GitHub因AI编码代理流量激增导致服务可靠性危机，Microsoft被迫将流量路由至竞争对手AWS；HPE扩展GreenLake平台推出代理AI运维框架。这些事件共同揭示了AI代理规模部署带来的基础设施挑战正从理论走向现实。

## 主要新闻

### AWS发布Context智能层：知识图谱自动从代理交互中学习

AWS宣布推出AWS Context，一个新的知识图谱服务，作为其AI代理上下文智能层的核心。与需要人工维护的传统知识图谱不同，AWS Context通过分析现有数据自动映射关系——包括数据表、列含义、数据源之间的关联和权威数据源的识别。它将语义搜索与图谱级推理相结合，在运行时为AI代理提供结构化的企业知识。同时发布的还有Amazon S3 Annotations（GA）和AWS Glue Data Catalog skill assets（预览版），三者共同构成AWS的上下文智能栈。AWS Context与Snowflake Horizon Context、Microsoft Fabric IQ和Pinecone Nexus等产品直接竞争，但其核心卖点是零集成摩擦——对已运行S3和Glue的企业来说无需额外数据迁移。

**Source:** [AWS enters the context layer race with a graph that learns from agents, not manual curation](https://venturebeat.com/data/aws-enters-the-context-layer-race-with-a-graph-that-learns-from-agents-not-manual-curation)

### GitHub AI代理危机：Microsoft被迫路由流量至AWS

Microsoft确认正将GitHub流量路由至最大竞争对手Amazon Web Services，原因是AI编码代理活动的空前激增超出了平台可靠性阈值。GitHub在2026年5月记录了9次服务降级事件，6月可用性持续低于99%，未能达到企业客户合同约定的"三个九"标准。核心问题是：AI编码代理的自动生成代码、提交和PR创建活动产生了远超传统人工操作的API调用量。Azure迁移进度不足以吸收激增流量，AWS作为临时缓冲提供溢出容量。GitHub CTO承认2月和3月均未达到企业SLA承诺，预计改善需到2026年9月。

**Source:** [GitHub's AI Agent Crisis Forces Microsoft to Tap AWS as Outages Break Enterprise SLAs](https://www.techtimes.com/articles/318481/20260616/githubs-ai-agent-crisis-forces-microsoft-tap-aws-outages-break-enterprise-slas.htm)

### HPE扩展GreenLake：推出代理AI运维与私有云管理

HPE宣布扩展GreenLake平台，推出GreenLake Intelligence——一个面向混合云和AI运维的代理AI框架。它包括AI代理的中央注册中心、编排工具和治理控制机制，可跨基础设施、应用和运维工作流管理代理。同时发布的HPE OpsRamp Operations Copilot为IT团队提供AI代理和LLM的可观测性，包括AI使用量监控、token消耗和运营成本追踪。HPE还更新了私有云产品线，新增Morpheus软件和标准化的气隙部署选项，支持VMware vSphere 9验证。

**Source:** [HPE expands GreenLake with agentic AI & private cloud](https://datacenternews.asia/story/hpe-expands-greenlake-with-agentic-ai-private-cloud)

### Kong发布Context Mesh：自动将企业API转化为AI代理工具

Kong宣布推出Kong Context Mesh，能够自动发现企业API、将其转化为代理可消费的工具，并以运行时治理方式部署这些工具。该产品利用Kong Konnect平台已有的端点、模式、认证要求和策略信息，自动生成符合Model Context Protocol的工具定义，并在Kong AI网关上强制执行策略。Context Mesh目前以技术预览版形式提供。

**Source:** [Kong Launches Context Mesh](https://www.devopsdigest.com/kong-launches-context-mesh)

## 分析

本周最引人注目的科技新闻无疑是GitHub的AI代理危机。这不仅是单一平台的服务故障，而是AI代理规模部署带来的结构性挑战的首次大规模显现。GitHub Actions和Copilot每天处理数以亿计的事件，当AI编码代理开始自主生成代码、创建PR、运行CI流水线时，平台的API调用模式发生了根本性转变——从人驱动（间歇性、可预测）变为机器驱动（持续性、爆发性）。Google向SpaceX支付每月9.2亿美元获取AI计算容量的消息从侧面印证了这一趋势的普遍性：即使是自建超大规模云基础设施的公司也无法独自满足AI代理带来的算力需求。

AWS Context的发布则标志着"上下文层"成为企业AI基础设施的新战场。Databricks的Genie Ontology、Snowflake的Horizon Context、Microsoft的Fabric IQ和AWS Context同台竞争，每家都在争夺AI代理的"企业知识控制权"。AWS的核心优势在于其S3/Glue/Lake Formation的用户基数，零数据迁移的承诺对已深度绑定AWS的企业极具吸引力。但这也意味着企业在选择AI平台时将面临更复杂的锁定考量。

HPE GreenLake Intelligence的更新则代表了传统IT基础设施厂商对AI代理浪潮的回应。将代理治理、运维监控和成本管理整合到统一的私有云管理平台中，对于大型企业的IT运维团队来说是一个务实的方向。

## 结论

2026年6月的科技新闻传递了一个清晰信号：AI代理的大规模部署正在重塑整个科技基础设施栈——从GitHub这样的开发者平台到AWS这样的云基础设施，再到HPE这样的企业IT供应商。GitHub危机表明平台工程团队需要重新审视CI/CD流水线的容量规划，AWS Context的出现意味着企业AI架构中"上下文层"将成为与计算、存储并列的核心组件。对于技术决策者而言，建立AI代理的可观测性、容量规划和成本管理机制，已从可选项变为必选项。
